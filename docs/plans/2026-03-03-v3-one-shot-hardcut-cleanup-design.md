# V3 一次性硬切无兼容清债设计

日期：2026-03-03  
状态：已定稿（与需求方逐节确认）

## 1. 目标与评分基线

本设计目标是以一次性硬切方式清除历史兼容负担、无效代码和扩展噪音，建立单轨 V3 运行时：规则集中、边界清晰、默认拒绝非标准输入，不再保留任何“为了迁就历史数据/历史入口”的回退逻辑。该方案优先追求长期可维护性与可演进性，而非短期兼容稳定。

当前基线评估（10 分制）：综合健康度 6.8、历史包袱残留 3.2、技术债压力 5.9、兼容性代价 7.6、解耦可扩展性 6.3。主要问题是兼容规则分散（Store/Domain/Embed 多点重复）、输入语义宽松（别名和兜底过多）、以及少量未消费导出/状态造成的维护噪音。

目标态评分：综合健康度 >= 8.8，兼容性代价 <= 2.0，解耦可扩展性 >= 8.5。实现方式不是“新增层层包装”，而是直接删除兼容路径、收紧协议、统一几何与校验规则、压缩无效表面。

硬约束：
- Node 策略为 `>=24`，允许 25。
- 运行时只接受严格 V3。
- 兼容回退不是暂缓，而是显式删除。

## 2. 一次性硬切的架构边界与删除策略

采用四层固定边界：`UI(展示/事件)` -> `Store(编排)` -> `Application(事务用例)` -> `Domain(纯规则)`。新增统一 `Geometry` 子域，承接命中测试与绘制几何规则，防止 Store 与 Canvas 双份逻辑漂移。

一次性删除与收敛内容：
- 删除对象类型历史别名映射，仅保留 `particle/electric-field/magnetic-field`。
- 删除 Embed 历史参数别名，只保留单一参数命名（不再接受 `m/scene_data/filename/material_id/mid/id/lang` 等）。
- 删除场景宽松接收语义：去掉 `passthrough`、去掉隐式字段容错，不再静默修复非标准载荷。
- 删除未消费接口与状态：`layoutMode`、`sceneNames`、`mapDragIntent`、`remove`、`listMockMaterials`、`getLatest`（若无真实消费则直接移除）。
- 删除 legacy 术语残留（lint 文案、注释、配置中的历史桥接措辞）。

工程一致性同步收口：
- Docker 基线与本地/CI 对齐 Node 24（CI 继续覆盖 24/25）。
- 文档中的协议说明仅描述硬切后的单轨语义，不保留“旧别名仍可用”的暗示。

该策略属于高强度手术，收益是后续变更不再受兼容分支拖累，代价是一次变更面较大，需要严格门禁验证。

## 3. 数据流、错误语义与验收门禁

重构后数据流定义为单向可追踪：UI 只产生命令意图，不直接写业务状态；Store 只负责生命周期与调用编排；Application 承担命令事务和版本冲突处理；Domain 负责确定性状态迁移；ReadModel 负责 UI 投影。Canvas 绘制与 hit-test 统一依赖 Geometry 规则源，禁止复制规则。

错误语义统一为有限集合：
- `validation`：输入/协议不合法；
- `version_conflict`：修订号冲突；
- `network`：远程场景获取失败；
- `runtime`：执行期异常；
- `invalid-command`：嵌入命令不支持。

Store 不再拼接多套错误文本，只透传标准错误码与信息，UI 决定展示文案。这样能避免跨层语义漂移和重复翻译。

验收门禁（必须同时满足）：
1. `npm run quality:all` 全绿。
2. `npm run build:frontend` 全绿。
3. 正向链路通过：create/edit/play/import/embed。
4. 负向链路新增并通过：旧 URL 别名、旧对象类型、宽松场景字段必须失败。
5. Node 策略校验通过：`engines` 为 `>=24`，CI 覆盖 24/25，Docker 对齐 24。

完成判定：兼容代码路径和未使用出口清零（或有明确保留理由），门禁全通过，文档与实现语义一致。
